# Как работает умный парсер — объяснение с нуля

> Этот файл объясняет технологии и архитектурные решения нового `scraper.py`.
> Написан для тебя — чтобы ты понял не только **что** сделано, но и **почему**.

---

## Проблема старого парсера

Старый код делал одно: `httpx.get(url)` → `BeautifulSoup` → убрать теги → вернуть текст.

Это работает только на ~30% сайтов. Почему?

Потому что современные сайты бывают двух типов:

| Тип | Пример | Как рендерится HTML |
|---|---|---|
| **SSR / Static** | Wikipedia, блоги, документация | Сервер сразу возвращает готовый HTML |
| **SPA (Single Page App)** | React/Vue/Angular сайты, HH.ru, Notion | Сервер возвращает пустой `<div id="root"></div>`, JS строит страницу в браузере |

Если сайт — SPA, то `httpx.get()` возвращает пустую страницу. JS не выполнялся.

---

## Решение: трёхуровневая стратегия

```
  URL входит
     │
     ▼
┌─────────────┐    много текста?
│  httpx GET  │ ─────────────────────────────► вернуть текст
└─────────────┘
      │ мало текста (<300 символов)
      ▼
┌──────────────────┐    текст появился?
│ Playwright       │ ──────────────────────────► вернуть текст
│ (браузер+JS)     │
└──────────────────┘
      │ всё равно пусто
      ▼
   вернуть None (сайт недоступен / заблокировал)
```

Плюс **краулинг**: после первой страницы парсим ещё до 4 внутренних ссылок того же домена.

## Технология 1: httpx

**Что это:** асинхронный HTTP-клиент для Python. Замена стандартному `requests`, но с поддержкой `async/await`.

**Почему async важен:** FastAPI работает на асинхронном event loop. Если использовать синхронный `requests.get()` внутри async-функции — он **блокирует весь сервер** на время запроса. Никакие другие пользователи не смогут подключиться.

```python
# ПЛОХО — блокирует event loop
import requests
def scrape(url):
    r = requests.get(url)  # все ждут

# ХОРОШО — не блокирует
import httpx
async def scrape(url):
    async with httpx.AsyncClient() as client:
        r = await client.get(url)  # сервер продолжает работать
```

**Используется для:** статических сайтов, Wikipedia, блогов, любой SSR-страницы.

## Технология 2: Playwright

**Что это:** инструмент для управления настоящим браузером (Chromium/Firefox/Safari) из кода. Изначально создан для тестирования, но идеально подходит для парсинга JS-сайтов.

**Как работает:** запускает реальный браузер в фоне (без окна — headless), открывает страницу, ждёт пока JS отрендерит DOM, возвращает итоговый HTML.

```python
async with async_playwright() as p:
    browser = await p.chromium.launch(headless=True)
    page = await browser.new_page()
    await page.goto("https://react-site.com", wait_until="load")
    await asyncio.sleep(1.5)          # дать JS время отрендериться
    html = await page.content()       # получить итоговый DOM
    await browser.close()
```

**wait_until параметр** — очень важная деталь:
- `"load"` — ждать пока загрузятся HTML + CSS + картинки. **Надёжный выбор.**
- `"networkidle"` — ждать пока все сетевые запросы утихнут. **Слишком строгий** — сложные SPA никогда не достигают тишины.
- `"domcontentloaded"` — только HTML, без ресурсов. **Слишком ранний** — JS не успел отработать.

**Используется для:** React/Vue/Angular сайтов, SPA, любых страниц с динамическим контентом.

## Технология 3: readability-lxml

**Что это:** Python-порт алгоритма Mozilla Readability — того самого, что стоит за режимом «Читалка» в Firefox.

**Проблема простого BeautifulSoup:** когда ты делаешь `soup.body.get_text()`, ты получаешь **весь** текст страницы — включая меню навигации, футер, виджеты, куки-баннеры, имена кнопок. Это мусор для RAG.

**Что делает Readability:** анализирует плотность текста в каждом блоке DOM, находит главный контентный блок (статью, описание продукта, документацию) и игнорирует всё остальное.

```python
from readability import Document

doc = Document(html)
summary_html = doc.summary()   # только главный контент, в HTML
# потом парсим этот чистый HTML через BeautifulSoup
soup = BeautifulSoup(summary_html, 'html.parser')
text = soup.get_text(separator=' ', strip=True)
```

**Используется для:** всех страниц — как замена ручному удалению тегов.

## Технология 4: Краулинг (обход страниц)

Одна страница часто не даёт полного контекста. Если у сайта есть страницы `/about`, `/pricing`, `/faq` — они тоже важны для RAG.

**Как работает наш краулер:**

1. Парсим первую страницу, извлекаем все ссылки `<a href="...">`.
2. Фильтруем: оставляем только ссылки **того же домена** (не уходим на внешние сайты).
3. Нормализуем URL: убираем `#fragment` и `?query` — чтобы не считать `/page#section1` и `/page#section2` разными страницами.
4. Обходим до `max_pages - 1` дополнительных страниц.
5. Между запросами пауза `0.5 сек` — это называется **polite crawling** (вежливый краулинг). Не нагружаем сервер.

```python
def _same_domain_links(html, base_url):
    base_netloc = urlparse(base_url).netloc
    for a in soup.find_all('a', href=True):
        href = urljoin(base_url, a['href'])        # абсолютный URL
        parsed = urlparse(href)
        if parsed.netloc != base_netloc: continue  # другой домен — пропуск
        clean = f"{scheme}://{netloc}{path}"       # убрать #fragment и ?query
        result.append(clean)
```

## Почему именно такой порядок (httpx → Playwright)?

**Playwright дорогой:** запускает целый браузер. Это ~500ms на старт + больше RAM. Если сайт простой — это бессмысленная трата ресурсов.

**httpx дешёвый:** простой TCP-запрос, 50-200ms, минимум памяти.

Поэтому: пробуем дешёвое → только если не получилось — дорогое.

Порог `_MIN_CHARS = 300` — эмпирическое значение. Если вернулось меньше 300 символов — скорее всего это пустой JS-шаблон.

## Структура итогового scraper.py

```
scraper.py
├── _html_to_text(html)          — readability → BeautifulSoup fallback
├── _looks_empty(text)           — проверка на JS-заглушку
├── _same_domain_links(html, url)— извлечение внутренних ссылок
├── _fetch_httpx(url)            — быстрый HTTP-запрос
├── _fetch_playwright(url)       — браузерный запрос с JS
├── _scrape_one(url)             — httpx → Playwright для одного URL
└── scrape_website(url)          — краулер: вызывает _scrape_one на N страницах
```

## Что ещё можно улучшить (если будет нужно)

| Улучшение | Зачем |
|---|---|
| `robots.txt` проверка | Уважать правила сайта (этика + юридика) |
| Ограничение по глубине | Не уходить на `/blog/tag/python/page/47` |
| Кеш уже спарсенных URL | Не парсить один домен дважды |
| PDF-парсинг (`pdfplumber`) | Многие компании хранят документацию в PDF |
| Sitemap.xml парсинг | Надёжнее чем следовать ссылкам — сайт сам говорит что у него есть |
| Прокси-ротация | Чтобы не получить бан при массовом парсинге |
